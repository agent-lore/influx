## Codebase Patterns
- Quality checks: `uv run ruff check src/ tests/`, `uv run ruff format --check src/ tests/`, `uv run pyright`, `uv run pytest` (make not available in this env)
- Exception hierarchy: `InfluxError` -> `ConfigError` -> `PromptValidationError` in `src/influx/errors.py`
- Build system: hatchling, dependencies in `pyproject.toml`, lock file `uv.lock`
- Python 3.12 strict (`>=3.12,<3.13`), uses `uv` for package management
- Source layout: `src/influx/` package with `py.typed` marker
- Tests in `tests/` directory, conftest at `tests/conftest.py`
- Unit tests go in `tests/unit/` subdirectory (created in US-002)
- FR-ST-2 slug pattern: `^[a-z0-9]+(-[a-z0-9]+)*$`, max 40 chars
- `pytest-cov` is not installed; coverage verification is by inspection/line count
- Prompt template variables use `{var_name}` syntax; `REQUIRED_VARIABLES` in `prompts.py` defines per-key sets
- `PromptValidationError` for template var errors, `ConfigError` for loading/structural errors
- Config schema root model: `AppConfig` in `config.py`; `load_config()` returns `AppConfig`
- `load_config(path=)` accepts optional explicit path; otherwise uses `find_config_path()` discovery
- `ValidationError` from pydantic is caught and wrapped as `ConfigError` in `load_config()`
- Minimal valid TOML needs `[prompts.filter]`, `[prompts.tier1_enrich]`, `[prompts.tier3_extract]` with `text` keys
- Pydantic validators that raise `ConfigError` (not `ValueError`) bypass pydantic's `ValidationError` wrapping and propagate directly
- RSS entry unknown fields and prompts unknown keys use `model_validator(mode="before")` with explicit `ConfigError`
- TOML `[providers.*]` and `[models.*]` map to `dict[str, ProviderConfig]` and `dict[str, ModelSlotConfig]`
- Env-var overrides in §19: INFLUX_ARCHIVE_DIR→storage.archive_dir, INFLUX_OTEL_ENABLED→telemetry.enabled, INFLUX_OTEL_CONSOLE_FALLBACK→telemetry.console_fallback, AGENT_ZERO_WEBHOOK_URL→notifications.webhook_url
- `_apply_env_overrides()` mutates the raw dict before `AppConfig.model_validate()` so pydantic still validates overridden values
- Profile name regex: `^[a-z][a-z0-9-]{0,31}$` — validated via `field_validator` on `ProfileConfig`, raises `ConfigError` directly
- Provider `api_key_env` validation runs post-`model_validate()` in `load_config()` via `_validate_provider_api_keys()`; skipped when `api_key_env = ''`
- RSS `source_tag` missing/invalid cases handled by pydantic's `Literal["rss","blog"]` + required field validation, wrapped as `ConfigError` in `load_config()`
- `influx.example.toml` at repo root; round-trip tests set fake API keys via `monkeypatch.setenv()` to pass provider validation
- conftest fixture `influx_config_env` uses a `test-provider` with `TEST_PROVIDER_API_KEY` to avoid requiring real provider keys
- CLI dispatcher in `main.py` uses `main(argv=)` parameter for testability; `__main__.py` calls `main()` with no args (defaults to `sys.argv[1:]`)
- `validate-config` subcommand prints `model_dump_json(indent=2)` for the effective config
